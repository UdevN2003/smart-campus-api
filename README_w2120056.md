## Part 1: Service Architecture & Setup

> Establishes the foundation of the API. Ensuring a robust initial configuration and implementing self-discoverable endpoints (HATEOAS) means the frontend and mobile teams can interact with the architecture smoothly, and the system can scale without breaking client applications.

### Q: Explain the default lifecycle of a JAX-RS Resource class.

By default, the JAX-RS runtime treats Resource classes as **per-request** — a brand new instance is created for every incoming HTTP request and destroyed once the response is sent.

**Impact on in-memory data management:**

- Instance variables **cannot** be used to store in-memory databases (Maps/Lists), as the data would be wiped on every request.
- Data structures must be declared as `static` or managed within a **separate Singleton data-store class**.
- Because multiple requests can arrive simultaneously, shared data structures must use **thread-safe collections** (e.g., `ConcurrentHashMap`) to prevent race conditions and data corruption.

---

### Q: Why is HATEOAS considered a hallmark of advanced RESTful design?

**HATEOAS** (Hypermedia as the Engine of Application State) is the highest level of REST maturity because it makes APIs **self-discoverable**. The server embeds navigational links directly within response payloads, indicating what actions or related resources are available next.

**Benefits over static documentation:**

| Static Documentation | HATEOAS |
|---|---|
| Frequently becomes outdated | Dynamically reflects current API state |
| Clients hardcode URLs | Clients follow server-provided links |
| URL changes break clients | Clients are decoupled from URL structure |

## Part 2: Room Management

> Defines the core physical entities of the campus, balancing network performance with client-side efficiency, while enforcing strict business logic to prevent data orphans when decommissioning rooms.

### Q: Returning only IDs vs. full room objects — what are the implications?

| Approach | Network Bandwidth | Client Processing |
|---|---|---|
| **IDs only** | Low (fast initial response) | High — triggers N+1 query problem |
| **Full objects** | Higher (larger JSON payload) | Low — all data in a single round-trip |

**Recommendation:** Return full objects when the client needs complete metadata, to minimize latency and additional round-trips.

---

### Q: Is the DELETE operation idempotent in this implementation?

**Yes.** Idempotency means multiple identical requests produce the same server state as a single request.

**Example flow for `DELETE /rooms/LIB-301`:**

1. **First request:** Room is found and removed → `204 No Content`
2. **Second request:** Room is already gone → `404 Not Found`

Although the HTTP status code differs, the **server state remains the same** (the room is still deleted). Because the state does not change on the second request, the operation is strictly idempotent.

## Part 3: Sensor Operations & Linking

> Handles hardware integration, enforces relational integrity (sensors only placed in valid rooms), and provides flexible query-based retrieval for facilities managers to filter sensors across campus.

### Q: What happens if a client sends data in the wrong format to a `@Consumes(APPLICATION_JSON)` endpoint?

The `@Consumes(MediaType.APPLICATION_JSON)` annotation tells the JAX-RS runtime exactly which data formats the endpoint can process.

If a client sends `text/plain` or `application/xml`:
- JAX-RS inspects the `Content-Type` header of the incoming request.
- The request is **intercepted and rejected before reaching the Java method**.
- The runtime automatically returns **HTTP 415 Unsupported Media Type**.

This protects the application from attempting to parse incompatible data formats.

---

### Q: Why is `@QueryParam` preferred over path-based filtering (e.g., `/sensors/type/CO2`)?

In RESTful architecture, **URL paths identify resources** while **query parameters modify or filter collections**.

| Criteria | `@QueryParam` | Path-based |
|---|---|---|
| **Optionality** | Optional by nature (`/sensors` still works) | Requires separate method per variation |
| **Stackability** | Easily combined: `?type=CO2&status=ACTIVE` | Rigid; order-dependent URL structure |
| **Semantics** | Correctly implies filtering a collection | Incorrectly implies a sub-resource exists |

## Part 4: Deep Nesting with Sub-Resources

> Managing historical time-series data for hardware readings requires complex URL routing. Sub-Resource Locators keep the codebase clean, modular, and maintainable as the project grows.

### Q: What are the architectural benefits of the Sub-Resource Locator pattern?

The primary benefit is enforcement of the **Single Responsibility Principle**.

**Without Sub-Resource Locators:**
- All nested paths (e.g., `sensors/{id}/readings`) are defined in one class.
- The class becomes a massive, unreadable **"God Class"**.
- Difficult to test, maintain, and causes merge conflicts in team environments.

**With Sub-Resource Locators:**
- Acts as a **router** — intercepts nested URL requests and delegates to focused, smaller classes.
- Sub-resource classes are instantiated with specific context (e.g., `sensorId` via constructor injection).
- Eliminates repetitive URL parameter extraction across every sub-method.

## Part 5: Advanced Error Handling & Logging

> Transforms the application into an enterprise-grade service. Hiding internal stack traces protects campus infrastructure from malicious actors, while accurate HTTP codes and comprehensive logging ensure full system observability.

### Q: Why is HTTP 422 more semantically accurate than 404 for a missing reference in a valid payload?

| Status Code | Meaning | When to Use |
|---|---|---|
| `404 Not Found` | The **endpoint/URL** does not exist | Route cannot be resolved |
| `422 Unprocessable Entity` | The endpoint exists and JSON is valid, but a **logical data error** prevents processing | Referenced entity (e.g., `roomId`) not found in the database |

When posting a sensor with a non-existent `roomId`, the endpoint `/api/v1/sensors` is valid and the JSON is well-formed — the issue is a **semantic/logical error inside the payload**, making `422` the correct response.

---

### Q: What are the cybersecurity risks of exposing Java stack traces?

Exposing stack traces is a vulnerability known as **Information Disclosure**. An attacker can harvest:

- **Framework names and version numbers** (e.g., Jersey, Tomcat) → search for known CVEs
- **Internal file paths** → craft directory traversal attacks
- **Database schema details** → targeted SQL injection
- **Internal Java class names** → reverse-engineer application structure

**Mitigation:** Use a global exception mapper to return sanitized, generic error messages to external consumers while logging full traces internally.

---

### Q: Why use JAX-RS filters for logging instead of manual `Logger.info()` calls?

Using a `@Provider` filter enforces **Separation of Concerns**:

| Manual Logging | Filter-based Logging |
|---|---|
| Business logic becomes cluttered | Logging is isolated in one place |
| Developers may forget to add logs | Every request/response is intercepted automatically |
| Inconsistent coverage across endpoints | **100% observability** guaranteed |
| Hard to update logging format globally | Single file to update |

A centrally managed filter intercepts every request and response, ensuring complete and consistent API observability without polluting business logic.
